## 1. Текущая архитектура: аудит и диагностика

### 1.1 Текущий пайплайн

Система состоит из 6 основных агентов, работающих последовательно с элементами параллелизма:

| # | Агент | Роль | LLM-вызовы | Слабое место |
|---|-------|------|------------|--------------|
| 1 | **Master Planner** | Анализ темы, определение языка и брендинга | 1 structured | Не исследует тему, не ищет факты |
| 2 | **Outline Agent** | Создание структуры слайдов | 1 structured | Одноразовый проход, нет итерации |
| 3 | **Writer Agent** | Написание контента для каждого слайда | N structured (батчи по 2) | Нет доступа к реальным данным, контент полностью галлюцинирован |
| 4 | **Theme Agent** | Создание визуальной темы | 0-1 (обычно пресет) | Ограничен пресетами, нет адаптации к контенту |
| 5 | **Layout Agent** | Выбор макета для каждого слайда | 1 structured | Не видит реальный рендер, решает "вслепую" |
| 6 | **HTML Composer** | Заполнение шаблонов данными | N text (батчи по 5) | Часто ошибается в формате данных |

Вспомогательные системы: QA Agent (структурная валидация + auto-fix + retry), Adaptive Sizing (подстройка шрифтов), Image Generation (до 5 слайдов), Markdown Inline (жирный/курсив).

### 1.2 Метрики текущего состояния

| Параметр | Значение |
|----------|----------|
| Шаблонов макетов | 18 (15 активных + 3 неиспользуемых) |
| Тем оформления | 10 пресетов |
| Тестов | 202 (10 файлов, 3055 строк) |
| Строк кода (server + client) | ~24,800 |
| Время генерации | ~60-120 секунд |
| LLM-вызовов на презентацию | ~15-25 |

### 1.3 Ключевые проблемы

**Проблема 1: Весь контент — галлюцинация.** Writer Agent генерирует текст исключительно из знаний LLM. Нет механизма поиска реальных фактов, статистики, цитат. Для бизнес-презентаций это критично — числа и факты должны быть достоверными или хотя бы правдоподобными.

**Проблема 2: Нет проверки визуального качества.** После рендеринга HTML никто не оценивает, как выглядит слайд. Layout Agent выбирает макет по описанию контента, но не видит результат. Переполнение текста, пустые области, нечитаемые графики — всё это обнаруживается только пользователем.

**Проблема 3: Одноразовая генерация без итерации.** Каждый агент делает один проход. Нет цикла "сгенерировать → оценить → улучшить". Anthropic в своей системе Research показали, что итеративный подход с обратной связью даёт на 90% лучшие результаты [1].

**Проблема 4: Нет нарративной связности между слайдами.** Writer получает контекст предыдущих слайдов (последние 4), но нет агента, который бы проверял общую историю: есть ли логические переходы, не повторяются ли аргументы, выстроена ли "красная нить" повествования.

**Проблема 5: Action Titles не применяются.** McKinsey-стиль требует, чтобы заголовок каждого слайда был законченным утверждением (action title), а не просто темой. Например, не "Рынок AI", а "Рынок AI вырастет в 3 раза к 2028 году" [2]. Текущая система этого не обеспечивает.

## 3. Направление A: Архитектура агентов

### A1. Итеративный цикл генерации (Generate → Evaluate → Refine)

**Текущее состояние:** Каждый агент делает один проход. QA Agent проверяет только структуру данных (наличие полей, количество буллетов), но не качество контента.

**Целевое состояние:** После каждого этапа — цикл оценки и доработки. Ключевой паттерн из Anthropic Research: "Subagents facilitate compression by operating in parallel with their own context windows... before condensing the most important tokens for the lead agent" [1].

**Реализация:**

```
Outline → Outline Critic → (fix if needed) → Writer → Content Critic → (fix if needed) → ...
```

Outline Critic проверяет: соблюдение Pyramid Principle, MECE-структуру аргументов, наличие action titles, баланс типов слайдов. Content Critic проверяет: плотность контента, отсутствие повторов между слайдами, наличие конкретных фактов/цифр, соответствие целевой аудитории.

**Оценка сложности:** Средняя. Добавляет 2-4 LLM-вызова, но значительно повышает качество.

### A2. Параллельная архитектура Writer Agent

**Текущее состояние:** Writer работает полу-последовательно (батчи по 2 слайда), передавая контекст предыдущих слайдов. Это медленно (5-8 секунд на слайд × 10 слайдов = 50-80 секунд).

**Целевое состояние:** Двухфазный подход, вдохновлённый Anthropic:

1. **Фаза 1 — Параллельная генерация:** Все слайды генерируются параллельно (батчи по 5), каждый Writer получает полный outline + описание всех слайдов как контекст.
2. **Фаза 2 — Последовательная когерентность:** Coherence Agent проходит по всем слайдам последовательно и правит переходы, убирает повторы, выстраивает нарратив.

Это сократит время Writer-фазы с ~60 секунд до ~15 секунд, при этом сохранив связность.

**Оценка сложности:** Средняя. Требует рефакторинга `runWriterParallel` и нового Coherence Agent.

### A3. Адаптивное масштабирование усилий

**Текущее состояние:** Одинаковый пайплайн для любой темы — 10 слайдов о "Котиках" проходят те же 6 агентов, что и 15 слайдов о "Стратегии цифровой трансформации предприятия".

**Целевое состояние:** Master Planner определяет уровень сложности и настраивает пайплайн:

| Уровень | Слайдов | Research | Итерации | Изображения |
|---------|---------|----------|----------|-------------|
| Простой | 5-8 | Нет | 0 | 1-2 |
| Стандартный | 8-12 | Базовый | 1 | 3-5 |
| Сложный | 12-20 | Глубокий | 2 | 5-8 |

Anthropic обнаружили, что "agents struggle to judge appropriate effort for different tasks, so we embedded scaling rules in the prompts" [1]. Тот же принцип применим к нашему пайплайну.

**Оценка сложности:** Низкая. Изменения в Master Planner и конфигурации пайплайна.

## 4. Направление B: Новые агенты и инструменты

### B1. Research Agent (Агент-исследователь)

**Проблема:** Все факты, цифры и примеры в презентациях — галлюцинации LLM. Для бизнес-контекста это неприемлемо.

**Решение:** Новый агент, который запускается после Outline и перед Writer. Получает список слайдов и для каждого ищет релевантные факты, статистику, цитаты.

**Инструменты Research Agent:**
- Web Search API (встроенный Data API) — поиск актуальных фактов и статистики
- Structured data extraction — извлечение числовых данных из найденных источников
- Citation formatter — форматирование источников для footnotes на слайдах

**Поток данных:**

```
Outline → Research Agent → Enriched Outline (с фактами) → Writer (использует факты)
```

Каждый слайд в outline получает дополнительное поле `research_data`:
```json
{
  "slide_number": 3,
  "title": "Рынок AI в 2026",
  "research_data": {
    "facts": ["Глобальный рынок AI достиг $184 млрд в 2024 (Statista)", ...],
    "statistics": [{"label": "CAGR", "value": "36.6%", "source": "Grand View Research"}],
    "quotes": []
  }
}
```

**Оценка сложности:** Высокая. Требует интеграции с поисковым API, парсинга результатов, валидации фактов. Но это самое импактное улучшение — превращает презентации из "красивых выдумок" в "обоснованные документы".

### B2. Design Critic Agent (Агент-критик дизайна)

**Проблема:** После рендеринга HTML никто не проверяет визуальное качество. Переполнение текста, пустые области, нечитаемые шрифты — всё это остаётся.

**Решение:** Агент, который получает отрендеренный HTML каждого слайда и оценивает:

| Критерий | Что проверяет | Как исправляет |
|----------|---------------|----------------|
| Переполнение | Текст выходит за границы | Сокращает текст или уменьшает шрифт |
| Пустота | Менее 40% площади занято | Добавляет контент или меняет макет |
| Контраст | Текст нечитаем на фоне | Корректирует цвета |
| Баланс | Элементы смещены в одну сторону | Перераспределяет layout |
| Консистентность | Разные стили между слайдами | Унифицирует |

**Реализация:** Можно использовать LLM с vision capabilities (передавая screenshot слайда) или чисто алгоритмический подход (анализ DOM-структуры, подсчёт символов, проверка CSS-свойств). Рекомендуется гибридный подход: алгоритмические проверки для быстрых случаев, LLM для сложных.

**Оценка сложности:** Средняя-высокая. Алгоритмическая часть проста, но LLM-vision требует дополнительной инфраструктуры.

### B3. Storytelling Agent (Агент-нарратор)

**Проблема:** Нарративная структура определяется на этапе Outline, но после написания контента никто не проверяет, сохранилась ли "красная нить" повествования.

**Решение:** Агент, работающий после Writer и перед Layout. Получает все написанные слайды и проверяет:

1. **Логические переходы:** Каждый слайд логически следует из предыдущего
2. **Pyramid Principle:** Главный вывод в начале, аргументы поддерживают его
3. **SCR-структура:** Ситуация → Проблема → Решение прослеживается
4. **Action Titles:** Каждый заголовок — законченное утверждение с выводом
5. **Нет повторов:** Каждый слайд добавляет новую информацию
6. **Эмоциональная дуга:** Есть нарастание, кульминация, разрешение

**Выход Storytelling Agent:**
- Исправленные заголовки (action titles)
- Предложения по переходам между слайдами (transition phrases)
- Оценка нарративной связности (0-100)
- Рекомендации по перестановке слайдов (если нужно)

**Оценка сложности:** Средняя. 1-2 LLM-вызова, но требует тщательного промпт-инжиниринга.

### B4. Speaker Coach Agent (Агент-тренер спикера)

**Проблема:** Speaker notes — второстепенный продукт, генерируемый Writer "заодно". Они часто повторяют текст слайда вместо того, чтобы дополнять его.

**Решение:** Отдельный агент, который после финализации контента создаёт профессиональные speaker notes:

- **Talking points:** 3-5 ключевых тезисов для каждого слайда
- **Transitions:** Фразы-переходы к следующему слайду ("А теперь давайте посмотрим на...")
- **Timing hints:** Рекомендуемое время на слайд
- **Audience engagement:** Вопросы к аудитории, паузы для реакции
- **Data context:** Дополнительный контекст для цифр на слайде

**Оценка сложности:** Низкая. 1 LLM-вызов на все слайды, минимальные изменения в пайплайне.

### B5. Data Visualization Agent (Агент визуализации данных)

**Проблема:** Текущая система поддерживает только базовые Chart.js графики (bar, line, pie). Нет инфографики, диаграмм процессов, mind maps, waterfall charts.

**Решение:** Специализированный агент, который анализирует данные слайда и выбирает оптимальный тип визуализации:

| Тип данных | Визуализация | Реализация |
|------------|-------------|------------|
| Сравнение 2-5 значений | Horizontal bar chart | Chart.js |
| Тренд во времени | Line/Area chart | Chart.js |
| Доли целого | Donut chart (не pie!) | Chart.js |
| Процесс/Flow | SVG-диаграмма | Custom SVG template |
| Иерархия | Treemap | D3.js или custom |
| Waterfall | Waterfall chart | Chart.js plugin |
| Gauge/KPI | Gauge chart | Custom SVG |
| Funnel | Funnel chart | Custom SVG |

**Оценка сложности:** Высокая. Требует новых шаблонов визуализации и интеграции дополнительных библиотек.

## 5. Направление C: Шаблоны и дизайн-система

### C1. Новые макеты слайдов

Текущие 18 макетов покрывают основные сценарии, но есть пробелы:

| Новый макет | Назначение | Приоритет |
|-------------|-----------|-----------|
| **big-number** | Один крупный KPI с контекстом | Высокий |
| **before-after** | Визуальное сравнение "до/после" | Высокий |
| **pyramid-diagram** | Иерархическая структура (3-5 уровней) | Средний |
| **swot-analysis** | Матрица 2×2 (SWOT, BCG, и т.д.) | Средний |
| **testimonial** | Отзыв клиента с фото и цитатой | Средний |
| **roadmap-visual** | Визуальный roadmap (не timeline) | Средний |
| **split-image-text** | 50/50 изображение + текст (вертикальный split) | Низкий |
| **three-column** | Три равные колонки (для сравнения 3 вариантов) | Низкий |
| **funnel** | Воронка (маркетинг, продажи) | Низкий |
| **dashboard** | Мини-дашборд с 4-6 метриками + графиком | Низкий |

### C2. Улучшение существующих макетов

**chart-slide:** Текущая реализация поддерживает только один график. Нужно добавить: комбинированные графики (bar + line), подписи данных, легенду, описание под графиком, выбор цветовой палитры из темы.

**icons-numbers:** Добавить варианты: горизонтальный (текущий), вертикальный (карточки в столбик), с прогресс-барами, с трендовыми стрелками (↑↓).

**timeline:** Добавить горизонтальный вариант (текущий только вертикальный), поддержку иконок для каждого события, цветовую кодировку по категориям.

### C3. Расширение системы тем

**Новые темы:**

| Тема | Стиль | Целевая аудитория |
|------|-------|-------------------|
| **Minimalist Mono** | Чёрно-белый, максимальный минимализм | Дизайнеры, креативные агентства |
| **Startup Neon** | Яркие неоновые акценты на тёмном фоне | Стартапы, tech pitch decks |
| **Academic Classic** | Serif шрифты, приглушённые цвета | Научные презентации |
| **Government Official** | Строгий, формальный, синий/серый | Государственные структуры |
| **Healthcare Clean** | Зелёный/белый, чистый, доверительный | Медицина, фармацевтика |

### C4. Система дизайн-токенов

Текущие CSS-переменные покрывают цвета и шрифты, но не хватает:

- **Spacing scale:** `--spacing-xs`, `--spacing-sm`, `--spacing-md`, `--spacing-lg`, `--spacing-xl`
- **Border radius scale:** `--radius-sm`, `--radius-md`, `--radius-lg`
- **Shadow scale:** `--shadow-sm`, `--shadow-md`, `--shadow-lg`
- **Animation tokens:** `--transition-fast`, `--transition-normal`
- **Typography scale:** `--text-xs`, `--text-sm`, `--text-base`, `--text-lg`, `--text-xl`, `--text-2xl`

Это позволит шаблонам быть более консистентными и упростит создание новых тем.

## 6. Направление D: Пользовательский опыт

### D1. Экспорт в PPTX

Самая запрашиваемая функция. Текущий HTML-вывод отлично работает для просмотра, но пользователям нужен PowerPoint для дальнейшего редактирования.

**Подход:** Использовать библиотеку `pptxgenjs` на сервере для конвертации структурированных данных слайдов (которые уже есть в `finalHtmlSlides`) в PPTX. Не конвертировать HTML→PPTX (это ненадёжно), а генерировать PPTX из тех же данных, что и HTML.

### D2. Шаринг и совместная работа

- Публичные ссылки на презентации (view-only)
- Встраивание (embed) через iframe
- QR-код для быстрого доступа с мобильных

### D3. Аналитика просмотров

- Сколько раз просмотрена презентация
- Среднее время на слайд
- На каком слайде зрители уходят
- Heatmap внимания

### D4. Пользовательские шаблоны

Позволить пользователям загружать свои CSS-темы или корпоративные стили:
- Загрузка логотипа компании
- Выбор корпоративных цветов
- Загрузка шрифтов
- Сохранение как "My Brand Kit"

| **D2. Шаринг** | ★★★☆☆ | Средняя | 🔵 P3 | 5 |
| **D3. Аналитика** | ★★☆☆☆ | Средняя | 🔵 P3 | 5 |
| **D4. Пользовательские шаблоны** | ★★★☆☆ | Высокая | 🔵 P3 | 5 |

### Рекомендуемый порядок реализации

**Спринт 1 (Максимальный ROI при минимальных затратах):**
- B3. Storytelling Agent — action titles + нарративная связность
- A1. Outline Critic — проверка структуры перед генерацией контента
- B4. Speaker Coach Agent — профессиональные заметки спикера

Эти три улучшения добавляют 3-5 LLM-вызовов, но кардинально повышают качество: заголовки становятся информативными, структура — логичной, заметки — полезными.

**Спринт 2 (Скорость и разнообразие):**
- A3. Адаптивное масштабирование — умный пайплайн
- A2. Параллельный Writer — ускорение генерации
- C1. Новые макеты — big-number, before-after, swot-analysis

**Спринт 3 (Данные и экспорт):**
- B1. Research Agent — реальные факты и статистика
- C2. Улучшенные графики
- D1. Экспорт в PPTX
