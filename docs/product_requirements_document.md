# Product Requirements Document (PRD): AI Presentation Generator

**Версия:** 1.0
**Дата:** 2026-02-23
**Автор:** Manus AI

---

## 1. Введение

Этот документ определяет требования к продукту (Product Requirements Document) для веб-приложения **AI Presentation Generator**. Цель проекта — создать интуитивно понятный, интерактивный инструмент для генерации презентаций с помощью AI, используя чат-интерфейс, аналогичный Manus и ChatGPT.

## 2. Целевая аудитория

- **Бизнес-аналитики и консультанты**, которым нужно быстро создавать структурированные презентации на основе данных.
- **Менеджеры по продуктам и маркетологи**, которым необходимо готовить питчи, отчеты и маркетинговые материалы.
- **Студенты и преподаватели**, использующие презентации для обучения и защиты проектов.

## 3. Ключевые возможности (Features)

- **Чат-интерфейс:** Все взаимодействие с AI происходит в формате диалога.
- **Генерация презентаций:** Создание полноценных презентаций (структура, контент, дизайн) по текстовому запросу.
- **Итеративная коррекция:** Возможность корректировать любой аспект презентации (текст, порядок слайдов, дизайн) через чат.
- **Прозрачность процесса:** Отображение статуса и текущего шага работы AI в реальном времени.
- **Интерактивные артефакты:** Просмотр и взаимодействие с генерируемыми файлами (документы, код, превью слайдов) прямо в интерфейсе.
- **Управление проектами:** Сохранение истории чатов и проектов для последующего доступа.

## 4. Техническая архитектура

### 4.1. Обзор

Приложение будет состоять из трёх основных компонентов: **Frontend**, **Backend** и **Движок v2.0**.

### 4.2. Технологический стек

| Компонент | Технология | Обоснование |
|:---|:---|:---|
| **Frontend** | React (с Vite) + TypeScript | Современный, быстрый и масштабируемый фреймворк для UI. TypeScript обеспечивает строгую типизацию. |
| **Стилизация** | TailwindCSS | Утилитарный CSS-фреймворк для быстрой и консистентной разработки. |
| **Backend** | FastAPI (Python) | Высокопроизводительный Python-фреймворк, идеально сочетается с движком, написанным на Python. |
| **База данных** | PostgreSQL | Надёжная и масштабируемая реляционная СУБД для хранения данных пользователей, чатов и проектов. |
| **Real-time** | WebSockets | Обеспечивает постоянное двунаправленное соединение для мгновенной доставки сообщений и стриминга. |
| **Кеширование** | Redis | Для управления сессиями и кеширования часто запрашиваемых данных. |

### 4.4. Компоненты

- **Frontend (React):** Отвечает за пользовательский интерфейс, взаимодействие с пользователем и отображение данных, получаемых от Backend.
- **Backend (FastAPI):** Является связующим звеном между Frontend и Движком. Управляет бизнес-логикой, аутентификацией, WebSocket-соединениями и взаимодействием с базой данных.
- **Движок v2.0 (Python):** Ядро системы, отвечающее за генерацию контента. Получает задачи от Backend и возвращает результаты.

### 4.5. Real-time коммуникация (WebSockets)

Процесс обеспечивает полную прозрачность для пользователя:
1. Frontend устанавливает WebSocket-соединение с Backend.
2. Пользователь отправляет запрос.
3. Backend запускает Движок v2.0 в фоновом режиме.
4. Движок генерирует события о прогрессе (`step_started`, `artifact_generated`).
5. Backend немедленно пересылает эти события на Frontend, который обновляет UI.

## 5. Пользовательский опыт и интерфейс (UX/UI)

### 5.1. Философия дизайна

Интерфейс должен быть **интуитивным, прозрачным и интерактивным**. Пользователь всегда должен понимать, что происходит, и иметь возможность вмешаться в процесс. Основная метафора — это **беседа с умным ассистентом** в рабочем пространстве, где создаются и обсуждаются реальные документы.

### 5.2. Структура интерфейса

Основной экран разделен на три части: **Список чатов/проектов (слева)**, **Основная область чата (в центре)** и **Панель артефактов (справа)**.

| Компонент | Описание | Ключевые элементы |
|:---|:---|:---|
| **Список проектов** | Панель слева для навигации между чатами/проектами. | - Кнопка "+ Новый чат"<br>- Список прошлых чатов с заголовками<br>- Поиск по истории |
| **Область чата** | Основная лента сообщений. | - Сообщения пользователя<br>- Сообщения AI<br>- **Карточки статуса** (например, "*Анализирую S1...*")<br>- **Карточки артефактов** (превью файлов)<br>- Поле ввода |
| **Панель артефактов** | Появляется справа, когда генерируется артефакт (по аналогии с Claude Artifacts). | - Live-превью для HTML/CSS<br>- Просмотрщик PDF/изображений<br>- Редактор кода<br>- Кнопки: "Скачать", "Открыть в новой вкладке", "Версии" |

### 5.3. Ключевой сценарий (User Flow)

1. Пользователь нажимает "+ Новый чат" и вводит запрос.
2. В чате появляется карточка статуса, отображающая прогресс выполнения задачи (`Планирую работу...`, `[x] S1: Анализ`, `[...] S2: Дизайн нарратива`).
3. По мере генерации артефактов (документов, слайдов) в чате появляются их превью.
4. При клике на превью артефакт открывается в **Панели артефактов** справа для детального просмотра и редактирования.
5. Пользователь может в любой момент написать в чат корректировку (например, *"Поменяй второй слайд, сделай его более строгим"*), и AI перегенерирует соответствующий артефакт, который обновится в панели.

## 6. Нефункциональные требования

- **Производительность:** Время ответа на простое сообщение не должно превышать 1 секунду. Генерация средней презентации (10-15 слайдов) должна занимать не более 2 минут.
- **Надежность:** Система должна быть доступна 99.9% времени.
- **Масштабируемость:** Архитектура должна поддерживать одновременную работу не менее 1000 пользователей.
- **Безопасность:** Все данные пользователей должны быть зашифрованы как при передаче, так и при хранении.

## 7. План развития (Roadmap)

- **MVP (Minimum Viable Product):** Реализация основного сценария генерации презентации с возможностью текстовой коррекции. Поддержка одного типа дизайна.
- **v1.1:** Добавление поддержки нескольких дизайн-систем и возможности их выбора.
- **v1.2:** Внедрение функции редактирования артефактов непосредственно в панели.
- **v2.0:** Поддержка командной работы и совместного редактирования проектов.
_# 8. Детальные сценарии (User Stories)_

_## 8.1. US-1: Успешная генерация презентации с нуля_

- **Как** Бизнес-аналитик
- **Я хочу** ввести текстовый запрос на создание презентации
- **Чтобы** получить готовую презентацию, соответствующую моему запросу, с минимальными усилиями.

**Шаги:**
1. Пользователь открывает приложение и нажимает «+ Новый чат».
2. Вводит в поле ввода: «Сделай презентацию на 10 слайдов о влиянии квантовых вычислений на финтех».
3. Нажимает «Отправить».
4. В чате появляется сообщение от AI: «Принято! Начинаю работу над вашей презентацией. Я буду держать вас в курсе процесса».
5. Появляется карточка статуса:
   - `[ ] S0: Планирование`
   - `[ ] S1: Анализ контекста`
   - `[ ] S2: Разработка нарратива`
   - `[ ] S3: Разработка дизайна`
   - `[ ] S4: Генерация контента`
   - `[ ] S5: Валидация`
6. Карточка статуса обновляется в реальном времени, отмечая выполненные шаги.
7. После шага S2 в чате появляется карточка артефакта: `structure.md`. При клике на нее справа открывается панель с Markdown-структурой презентации.
8. После шага S4 в чате появляется карточка артефакта: `presentation.html`. При клике на нее в панели справа отображается интерактивное превью первого слайда.
9. После шага S5 AI пишет в чат: «Готово! Ваша презентация создана. Вы можете просмотреть ее в панели справа или скачать. Что-нибудь еще?»

_## 8.2. US-2: Коррекция контента слайда_

- **Как** Менеджер по продукту
- **Я хочу** попросить AI изменить текст на конкретном слайде
- **Чтобы** быстро внести правки, не редактируя файл вручную.

**Шаги:**
1. Презентация `presentation.html` открыта в панели артефактов.
2. Пользователь пишет в чат: «На слайде 3 убери упоминание про блокчейн и добавь больше про кибербезопасность».
3. AI отвечает: «Понял, вношу изменения в слайд 3».
4. Карточка статуса обновляется, показывая активность на шагах S4 и S5.
5. Превью в панели артефактов обновляется, показывая измененный слайд 3.
6. AI пишет: «Готово, я обновил слайд. Посмотрите, пожалуйста».

_## 8.3. US-3: Загрузка своего файла для анализа_

- **Как** Консультант
- **Я хочу** загрузить свой документ (например, годовой отчет)
- **Чтобы** AI использовал его как основу для создания презентации.

**Шаги:**
1. Пользователь нажимает «+» в поле ввода и выбирает файл `annual_report_2025.pdf`.
2. После загрузки файла пишет в чат: «На основе этого отчета сделай презентацию для совета директоров. Ключевые моменты: рост выручки, выход на новые рынки и планы на следующий год».
3. AI отвечает: «Отлично, я проанализирую документ и подготовлю презентацию».
4. Дальнейший процесс аналогичен US-1, но на шаге S1 AI использует предоставленный файл как основной источник информации.

# 9. Протокол WebSocket и Модели данных

## 9.1. WebSocket Messages

Все сообщения между клиентом и сервером передаются в формате JSON. Каждое сообщение имеет `type` и `payload`.

### Сообщения от Клиента (Frontend -> Backend)

- **`user_message`**: Отправка нового сообщения от пользователя.
  ```json
  {
    "type": "user_message",
    "payload": {
      "text": "Сделай презентацию...",
      "attachments": [{"file_id": "..."}]
    }
  }
  ```

- **`artifact_feedback`**: Отправка фидбека или запроса на изменение артефакта.
  ```json
  {
    "type": "artifact_feedback",
    "payload": {
      "artifact_id": "...",
      "feedback_text": "Поменяй заголовок на этом слайде"
    }
  }
  ```

### Сообщения от Сервера (Backend -> Frontend)

- **`ai_message`**: Обычное текстовое сообщение от AI.
  ```json
  {
    "type": "ai_message",
    "payload": {
      "text": "Начинаю работу..."
    }
  }
  ```

- **`status_update`**: Обновление статуса выполнения задачи.
  ```json
  {
    "type": "status_update",
    "payload": {
      "steps": [
        {"name": "S0: Планирование", "status": "completed"},
        {"name": "S1: Анализ", "status": "in_progress"},
        ...
      ]
    }
  }
  ```

- **`artifact_generated`**: Уведомление о создании нового артефакта.
  ```json
  {
    "type": "artifact_generated",
    "payload": {
      "artifact_id": "...",
      "filename": "presentation.html",
      "file_type": "html",
      "preview_url": "..."
    }
  }
  ```

## 9.2. Модели данных (Pydantic / PostgreSQL)

- **`User`**: Модель пользователя.
  - `id`, `username`, `email`, `password_hash`, `created_at`

- **`Project` / `Chat`**: Модель чата или проекта.
  - `id`, `user_id`, `title`, `created_at`

- **`Message`**: Модель сообщения в чате.
  - `id`, `chat_id`, `sender` (`user` или `ai`), `text`, `attachments`, `created_at`

- **`Artifact`**: Модель сгенерированного артефакта.
  - `id`, `chat_id`, `filename`, `file_type`, `storage_path`, `version`, `created_at`

## 10. Редактирование артефактов

Ключевой особенностью приложения является возможность не только просматривать, но и редактировать сгенерированные артефакты прямо в интерфейсе. Это позволяет пользователю осуществлять полный цикл работы над документом, не покидая приложение.

### 10.1. Редактирование текстовых файлов (Markdown, Code)

- **Механизм:** Когда пользователь открывает текстовый артефакт (`.md`, `.py`, `.json` и т.д.) в **Панели артефактов**, он отображается в текстовом редакторе (например, Monaco Editor).
- **Сохранение:** Пользователь вносит изменения и нажимает кнопку «Сохранить» (или `Ctrl+S`).
- **Уведомление AI:** Frontend отправляет на Backend WebSocket-сообщение `artifact_updated` с `artifact_id` и новым содержимым файла.
- **Реакция AI:** Backend передает эту информацию Движку. Движок может:
    1. **Просто принять изменения:** Если это не влияет на общую структуру (например, исправление опечатки).
    2. **Перезапустить часть пайплайна:** Если изменения существенны (например, изменение структуры в `structure.md`), AI может предложить перегенерировать зависимые артефакты.

### 10.2. Редактирование презентации (HTML)

Редактирование HTML-презентации — более сложная задача. Предлагается два подхода:

**Подход А (MVP): Редактирование через Markdown**

1. **Источник правды:** Основным редактируемым файлом является `structure.md` (или аналогичный файл с контентом).
2. **Процесс:** Пользователь редактирует текст, заголовки или порядок секций в `structure.md`.
3. **Перегенерация:** После сохранения `structure.md` AI автоматически запускает шаги S3-S4, чтобы перегенерировать `presentation.html` с учетом новых данных.
4. **Преимущества:** Простота реализации, сохранение консистентности.
5. **Недостатки:** Нельзя напрямую редактировать визуальные элементы на слайде.

**Подход Б (v1.2): Прямое редактирование слайдов (WYSIWYG)**

1. **Интерактивный редактор:** В **Панели артефактов** `presentation.html` отображается в интерактивном режиме. Текстовые блоки на слайде можно кликнуть и редактировать на месте.
2. **Сохранение изменений:** После редактирования текста на слайде пользователь нажимает «Сохранить».
3. **Синхронизация:** Frontend отправляет изменения на Backend. Backend обновляет как `presentation.html`, так и исходный `structure.md`, чтобы поддерживать их в синхронизированном состоянии.
4. **Преимущества:** Интуитивно и удобно для пользователя.
5. **Недостатки:** Значительно сложнее в реализации, требует двусторонней синхронизации данных.

**Рекомендация:** Начать с **Подхода А** для MVP, а **Подход Б** реализовать в последующих версиях.

### 10.3. Обновленный User Story (US-4)

- **Как** Бизнес-аналитик
- **Я хочу** открыть сгенерированный Markdown-файл со структурой презентации и отредактировать его прямо в приложении
- **Чтобы** быстро изменить порядок слайдов и их заголовки, и увидеть, как обновилась финальная презентация.

1. В чате сгенерирован артефакт `structure.md`. Пользователь кликает на него.

3. Пользователь меняет местами 3 и 4 слайд и исправляет заголовок 5-го слайда.
4. Нажимает «Сохранить».
5. AI пишет в чат: «Вижу, вы обновили структуру. Перегенерирую презентацию, чтобы отразить изменения».
6. Карточка статуса показывает активность на шагах S3-S5.
7. Превью `presentation.html` в панели артефактов обновляется с новым порядком и заголовками слайдов.
