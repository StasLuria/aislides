# ADR-001: Выбор архитектуры движка

**Дата:** 2026-02-23
**Статус:** Принято
**Авторы:** Stas, Manus AI

## Контекст

Необходимо выбрать архитектуру для движка генерации презентаций. Движок должен преобразовывать пользовательский запрос в готовую HTML-презентацию через многошаговый конвейер, управляемый LLM.

Рассматривались три подхода:

| Подход | Описание | Плюсы | Минусы |
|:---|:---|:---|:---|
| **Жёсткий pipeline** | Фиксированная последовательность S1→S2→S3→S4→S5 | Простота, предсказуемость | Не поддерживает частичную перегенерацию, ручные правки |
| **Полностью автономный агент** | LLM сам решает, что делать на каждом шаге | Максимальная гибкость | Непредсказуемость, сложность отладки, высокая стоимость |
| **LLM-планировщик + детерминированное исполнение** | LLM генерирует план один раз, RuntimeAgent исполняет его детерминированно | Гибкость + предсказуемость, поддержка правок | Сложнее жёсткого pipeline |

## Решение

Выбран подход **«LLM-планировщик + детерминированное исполнение»** (вариант 3).

Архитектура состоит из:

1. **S0_PlannerNode** — LLM генерирует JSON-план с указанием шагов и их параметров.
2. **PlanValidatorNode** — детерминированная проверка плана (существуют ли узлы, корректны ли зависимости).
3. **RuntimeAgent** — последовательное исполнение шагов плана с трансляцией событий через EventBus.

Все вызовы LLM проходят через библиотеку **Instructor**, которая гарантирует получение валидированных Pydantic-моделей и автоматические повторные попытки.

## Последствия

**Положительные:**
- Поддержка ручных правок (apply_edit) — S0 генерирует частичный план только для зависимых артефактов.
- Поддержка отмены (cancel) — RuntimeAgent проверяет cancel_token перед каждым шагом.
- Наблюдаемость — EventBus транслирует события для real-time UI.
- Тестируемость — каждый узел тестируется в изоляции с mock LLM.

**Отрицательные:**
- Качество плана зависит от качества промпта для S0.
- Дополнительная сложность по сравнению с жёстким pipeline.
