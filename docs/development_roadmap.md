## 1. Как пользоваться этим документом

Этот роадмап — **живой чек-лист**. После завершения каждой задачи разработчик обязан:

1. Заменить `☐` на `✅` напротив выполненной задачи.
2. Добавить дату завершения в колонку «Комментарий».
3. Закоммитить изменение: `docs: update roadmap — mark task X as done`.

Если задача оказалась сложнее, чем ожидалось, или потребовала изменения архитектуры — создать ADR в `docs/adr/` и добавить ссылку в колонку «Комментарий».

## 8. Мои предложения по защите от «сбиться с пути»

Помимо стандартных практик (тесты, CI, документация), я предлагаю следующие механизмы:

### 8.1. Правило «Спринт-отчёт»

После завершения каждого спринта разработчик пишет короткий отчёт (3-5 предложений) в `CHANGELOG.md`:

- Что было сделано?
- Какие проблемы возникли?
- Какие решения были приняты (ссылка на ADR)?

Это создаёт «хлебные крошки», по которым любой новый разработчик может восстановить историю проекта.

### 8.2. Правило «Контекстный файл» (CONTEXT.md)

В корне проекта лежит файл `CONTEXT.md`, который содержит:

- Текущий спринт и его цель.
- Список «в работе» задач.
- Известные проблемы и блокеры.
- Последнее принятое архитектурное решение.

Этот файл обновляется **каждый рабочий день**. Он позволяет любому разработчику за 2 минуты понять, «что сейчас происходит».

### 8.3. Правило «Smoke Test»

Перед каждым мёржем в `main` запускается автоматический smoke-тест, который проверяет:

- Все unit-тесты проходят.
- Все integration-тесты проходят.
- Линтер и type-checker не находят ошибок.
- Покрытие кода не упало ниже 90%.

Это гарантирует, что `main` всегда в рабочем состоянии.

### 8.4. Правило «Промпт как код»

Все промпты для LLM хранятся как строковые константы в отдельных файлах (`tools/prompts/`) и покрываются тестами. Изменение промпта — это такой же PR, как изменение кода. Это предотвращает «тихую деградацию» качества генерации.

### 8.5. Правило «Один PR — одна задача»

Каждый Pull Request решает ровно одну задачу из роадмапа. Это упрощает ревью, откат и понимание истории изменений.
