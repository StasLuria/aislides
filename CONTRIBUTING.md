# Правила разработки (How to Contribute)

**Версия:** 1.0
**Дата:** 2026-02-23

## 1. Философия

Мы стремимся к созданию высококачественного, поддерживаемого и хорошо документированного кода. Эти правила — не бюрократия, а инструмент для достижения этой цели. **Качество важнее скорости.**

## 2. Процесс работы над задачей (Definition of Done)

Задача считается **«Готовой» (Done)** только тогда, когда выполнены все следующие пункты:

1.  **Код написан** и соответствует стайлгайду.
2.  **Код покрыт тестами** (Unit + Integration), покрытие не ниже 90%.
3.  **Все тесты успешно проходят** в CI.
4.  **Код задокументирован** (docstrings для всех public-методов и классов).
5.  **Создан или обновлен ADR**, если было принято важное архитектурное решение.
6.  **Обновлен `CHANGELOG.md`** с описанием изменений.
7.  **Обновлен `development_roadmap.md`** — отмечен выполненный пункт.
8.  **Pull Request (PR) прошел ревью** и был смёржен в `main`.

## 3. Git Workflow

- **Основная ветка:** `main`.
- **Прямые коммиты в `main` запрещены.**
- **Модель ветвления:** `feature/JIRA-123-short-description` (например, `feature/PROJ-42-implement-runtime-agent`).
- **Коммиты:** Следуйте формату [Conventional Commits](https://www.conventionalcommits.org/).
  - `feat:` (новая фича)
  - `fix:` (исправление бага)
  - `docs:` (изменения в документации)
  - `style:` (форматирование, без изменения кода)
  - `refactor:` (рефакторинг без изменения функциональности)
  - `test:` (добавление или исправление тестов)
  - `chore:` (рутинные задачи, сборка)

## 4. Стандарты кода

- **Язык:** Python 3.11+
- **Стиль:** PEP 8.
- **Линтер:** `ruff` (конфигурация в `pyproject.toml`).
- **Форматирование:** `black`.
- **Проверка типов:** `mypy`.
- **Docstrings:** Google Style.

Все эти инструменты запускаются автоматически через pre-commit хуки и в CI.

## 5. Тестирование

- **Фреймворк:** `pytest`.
- **Покрытие:** `pytest-cov`, минимальный порог — 90%.
- **Типы тестов:**
  - **Unit-тесты:** Лежат рядом с кодом (`tests/unit/`). Тестируют один класс или функцию в изоляции. Все внешние зависимости (LLM, БД) должны быть замоканы.
  - **Integration-тесты:** Лежат в `tests/integration/`. Тестируют взаимодействие нескольких компонентов (например, `Planner` + `RuntimeAgent`). Могут использовать реальные, но контролируемые зависимости (например, тестовую БД).

## 6. Документация

Документация — это часть продукта. Она должна быть всегда актуальной.

- **Docstrings:** Каждый public-класс, метод и функция должны иметь docstring, объясняющий *что* он делает, его *аргументы* и *что* он возвращает.
- **`README.md`:** Обновляется после каждого спринта, отражая текущее состояние проекта и как его запустить.
- **`ONBOARDING.md`:** Содержит пошаговую инструкцию для нового разработчика.
- **`docs/adr/`:** Все важные архитектурные решения фиксируются здесь.
- **`CHANGELOG.md`:** Ведется автоматически на основе Conventional Commits.

## 7. Правила для промптов (Prompt Engineering)

Промпты для LLM — критическая часть системы. К ним применяются те же стандарты качества, что и к коду.

- **Хранение:** Все промпты хранятся как строковые константы в `tools/prompts/`. Один файл — один Node (например, `tools/prompts/s1_prompt.py`).
- **Формат:** Каждый промпт-файл содержит `SYSTEM_PROMPT` и `USER_PROMPT_TEMPLATE` как отдельные переменные.
- **Версионирование:** Изменение промпта — это полноценный PR с описанием причины изменения и ожидаемого эффекта.
- **Тестирование:** Каждый промпт покрыт тестом, который проверяет, что шаблон корректно форматируется с тестовыми данными и содержит все необходимые инструкции.
- **Ревью:** При ревью промпта проверяется: (1) чёткость инструкций, (2) наличие формата ответа, (3) наличие примеров, (4) отсутствие противоречий.

## 8. Файл CONTEXT.md

В корне проекта лежит файл `CONTEXT.md`, который обновляется ежедневно и содержит:

- Текущий спринт и его цель.
- Список задач «в работе».
- Известные проблемы и блокеры.
- Последнее принятое архитектурное решение.

Этот файл позволяет любому разработчику за 2 минуты понять текущее состояние проекта.


## 9. Чеклист перед коммитом (Pre-commit Checklist)

Перед каждым коммитом в `feature/...` ветку, убедитесь, что вы выполнили все пункты из этого чеклиста. Это гарантирует соблюдение Definition of Done.

| Статус | Шаг | Описание |
|:---:|:---|:---|
| ☐ | **Задача из Roadmap** | Убедитесь, что ваша работа соответствует задаче из `DEVELOPMENT_PLAN.md`. |
| ☐ | **Feature Branch** | Вы работаете в ветке `feature/...`, а не в `main`. |
| ☐ | **Тесты написаны и проходят** | Все тесты (`pytest tests/`) проходят, покрытие ≥ 90%. |
| ☐ | **`CHANGELOG.md` обновлён** | Добавлена запись о ваших изменениях в раздел `[Unreleased]`. |
| ☐ | **`CONTEXT.md` обновлён** | Файл отражает текущее состояние проекта. |
| ☐ | **`DEVELOPMENT_PLAN.md` обновлён** | Отмечен прогресс по вашей задаче (если применимо). |
| ☐ | **ADR создан** | Если было принято важное архитектурное решение, создан новый ADR в `docs/adr/`. |
| ☐ | **PR готов к ревью** | Ваш PR содержит ссылку на задачу, описание изменений и готов к ревью. |
